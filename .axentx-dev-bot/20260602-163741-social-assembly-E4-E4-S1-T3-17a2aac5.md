**Feature Proposal: Real‑time Collaboration Dashboard**

---

### 1. Objective
Provide users with a live, interactive workspace where team members can view, edit, and comment on shared assets (documents, diagrams, code snippets) simultaneously, reducing context‑switching and accelerating decision‑making cycles.

### 2. User Stories
- **As a product manager**, I want to see live updates to a roadmap document so I can align with engineering without waiting for refreshes.
- **As a designer**, I need to co‑edit wireframes with teammates and see cursor positions to avoid conflicts.
- **As a developer**, I want to paste code blocks into the dashboard and have syntax highlighting applied instantly for peer review.
- **As a stakeholder**, I require read‑only access with the ability to leave threaded comments that notify relevant owners.

### 3. Acceptance Criteria
| # | Criteria | Test Method |
|---|----------|-------------|
| AC1 | Multiple users can open the same asset and see each other's edits within ≤200 ms latency. | Load test with 10 concurrent clients; measure round‑trip time via WebSocket ping. |
| AC2 | Cursor presence indicators (name + color) are displayed for each active participant. | UI inspection; verify visibility when ≥2 users edit. |
| AC3 | Changes are persisted to the backend and survive page reloads. | Edit asset, reload browser, confirm edits remain. |
| AC4 | Comment threads support @mentions, email notifications, and resolution status. | Create comment, mention user, check email and UI status change. |
| AC5 | Permission model enforces read‑only, comment‑only, and edit roles per asset. | Attempt actions with each role; verify allowed/denied outcomes. |
| AC6 | The dashboard degrades gracefully to a read‑only view when WebSocket connection fails. | Disconnect network; ensure UI shows offline banner and disables edit controls. |

### 4. Technical Design
- **Frontend**: React 18 with Redux Toolkit for state management; use `yjs` CRDT library for conflict‑free collaborative editing; integrate Monaco Editor for code blocks and Fabric.js for canvas‑based assets.
- **Backend**: Node.js 20 + Express; WebSocket server via `ws`; persistence layer PostgreSQL with JSONB asset storage; Redis pub/sub for broadcasting presence events.
- **APIs**:
  - `GET /assets/:id` – retrieve asset metadata and initial content.
  - `POST /assets/:id/ops` – apply operational transform (handled by yjs provider).
  - `POST /assets/:id/comments` – create comment; triggers email via SendGrid.
  - `WS /collab/:assetId` – bidirectional stream for ops and presence.
- **Security**: JWT‑based authentication; role‑based access checked on each WebSocket message; sanitize all user‑generated content with DOMPurify; rate‑limit WebSocket messages (≤30 msg/s per connection).
- **Scalability**: Stateless WebSocket servers behind a sticky‑session load balancer (NGINX); horizontal pod autoscaling in Kubernetes based on concurrent connections.

### 5. Dependencies
- External: `yjs`, `monaco-editor`, `fabric.js`, `ws`, `redis`, `postgresql`, `sendgrid`.
- Internal: Existing user‑auth service, asset‑storage micro‑service, notification service.

## 1. Acceptance Criteria  
- **Tag Storage**  
  - Each `Community` document must have a `tags` field that is an array of strings.  
  - Tags are stored in lowercase, trimmed, and deduplicated.  
- **Tag Validation**  
  - Tags must be non‑empty strings, max length 30 characters, and match `/^[a-z0-9]+(?:-[a-z0-9]+)*$/`.  
  - Attempts to add invalid tags throw a `ValidationError`.  
- **Search by Tags**  
  - The discover API (`GET /communities`) accepts a `tags` query param (comma‑separated).  
  - The query returns only communities that contain **all** specified tags.  
  - Search is case‑insensitive.  
- **Pagination**  
  - The discover API supports `page` and `limit` query params.  
  - Results are sorted by `createdAt` descending.  
  - Response includes `total`, `page`, `limit`, and `items`.  
- **Security**  
  - No injection vulnerabilities: tags are sanitized and indexed.
