**1. Acceptance Criteria**

- **AC1:** A moderation rule can be created with a non‑empty `title`, optional `description`, and a `severity` value limited to the enum `["low","medium","high"]`. The created record must include a `version` field set to `1` and a `createdAt` timestamp (ISO‑8601) generated by the server.
- **AC2:** When a rule is edited, a new version is created: the `version` field increments by 1, `updatedAt` reflects the edit time, and the previous version remains immutable in the database.
- **AC3:** Archived rules are marked with `archived: true`, retain their latest version number, and are excluded from the “active rules” list shown to end‑users.
- **AC4:** The API endpoint `GET /api/rules` returns only active rules, sorted by `severity` (high → low) and then by `createdAt` descending. Each rule object includes `id`, `title`, `description`, `severity`, `version`, `createdAt`, and `updatedAt`.
- **AC5:** The API endpoint `GET /api/rules/:id/history` returns an array of all versions of the rule identified by `id`, ordered by `version` ascending, each entry containing `version`, `title`, `description`, `severity`, `createdAt`, `updatedAt`.
- **AC6:** Attempting to create or edit a rule with an invalid `severity` or missing `title` returns HTTP 400 with a descriptive error message.
- **AC7:** All timestamps (`createdAt`, `updatedAt`) are stored in UTC and are not future‑dated (i.e., `timestamp ≤ now` at the moment of persistence).
