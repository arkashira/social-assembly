**1. Acceptance Criteria**  
- **AC1 – Creation**: A moderation rule can be created with a non‑empty `title`, optional `description`, and a `severity` value limited to the enum `["low","medium","high"]`. On creation the rule must contain a `version` field set to `1`, a `createdAt` timestamp (UTC, ≤ now), and an `updatedAt` timestamp (same as `createdAt`).  
- **AC2 – Editing**: When a rule is edited, a new document is created with `version` incremented by `1`, `updatedAt` set to the edit time (UTC, ≤ now), and the previous version remains immutable.  
- **AC3 – Archiving**: A moderator can archive a rule; the rule is marked with `archived: true`, retains its latest `version`, and is excluded from the active rules list.  
- **AC4 – Retrieval of Active Rules**: `GET /api/rules` returns only active (non‑archived) rules, sorted by `severity` (high → medium → low) then by `createdAt` descending. Each rule object includes `id`, `title`, `description`, `severity`, `version`, `createdAt`, `updatedAt`.  
- **AC5 – Rule History**: `GET /api/rules/:id/history` returns an array of all versions of the rule identified by `id`, ordered by `version` ascending, each entry containing `version`, `title`, `description`, `severity`, `createdAt`, `updatedAt`.  
- **AC6 – Validation**: Attempts to create or edit a rule with an invalid `severity` or missing `title` return HTTP 400 with a descriptive error message.  
- **AC7 – Timestamp Integrity**: All timestamps (`createdAt`, `updatedAt`) are stored in UTC and are never future‑dated (`timestamp ≤ now` at persistence time).
